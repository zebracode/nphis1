# การวางระบบ POC: ระบบย่อย 2, 3, 4 ให้ครบ Loop การใช้งาน

จากเอกสาร `doc/req` วิเคราะห์และออกแบบการทำ POC เฉพาะส่วน **2. ระบบการตรวจรักษา**, **3. ระบบห้องเรียนออนไลน์**, **4. ระบบพอร์ทัลเกษตรกร** ให้เห็นรูปแบบการใช้งานเบื้องต้นครบวงจร (End-to-End Loop)

---

## 1. ภาพรวม Loop การใช้งาน (User Journey)

เพื่อให้ POC แสดง “การทำงานจริง” แบบครบ loop แนะนำให้เชื่อม 3 ระบบเป็น **หนึ่งวงจรการให้บริการ** ดังนี้:

```
[เกษตรกร] พอร์ทัลเกษตรกร
    │
    ├─ ลงทะเบียนฟาร์ม / สัตว์
    ├─ บันทึกข้อมูลประจำวัน (อาหาร, อาการ, น้ำหนัก, นม)
    └─ ขอนัดตรวจ (ปกติ / ฉุกเฉิน)
              │
              ▼
[โรงพยาบาล] ระบบการตรวจรักษา
    │
    ├─ รับเคสจากนัด / ลงพื้นที่ฟาร์ม
    ├─ บันทึกการตรวจรักษา (แยกตามคลินิก)
    ├─ ถ่ายภาพ/อัปโหลดภาพอาการ
    └─ (เลือกได้) มอบเคสให้นักศึกษาเพื่อฝึกปฏิบัติ
              │
              ▼
[อาจารย์/นักศึกษา] ห้องเรียนออนไลน์
    │
    ├─ นักศึกษา: ลงทะเบียนรับเคส, ดูประวัติ, บันทึกผล (ตามสิทธิ์)
    ├─ อาจารย์: อนุมัติ/แก้ไข/ตรวจสอบเคส
    └─ ระบบ: บันทึกต้นทุนการรักษาจริง (แม้ยกเว้นค่าใช้จ่ายเพื่อการสอน)
              │
              ▼
    สรุปผล / แจ้งเตือนกลับไปยังเกษตรกร (นัดครั้งถัดไป, คำแนะนำ)
```

**จุดสำคัญ:** เกษตรกรเป็น “จุดเริ่มต้น” (ลงทะเบียน + บันทึกข้อมูล + ขอนัด) → โรงพยาบาลเป็น “จุดกลาง” (ตรวจรักษา + ถ่ายภาพ + บันทึก) → ห้องเรียนเป็น “จุดสอนและควบคุมคุณภาพ” (นักศึกษาฝึก, อาจารย์อนุมัติ, บันทึกต้นทุน) → จากนั้นระบบสามารถปิด loop ด้วยการแจ้งเตือน/นัดครั้งถัดไปให้เกษตรกร

---

## 2. กลุ่มผู้ใช้งานและบทบาท (Roles) สำหรับ POC

| Role | ระบบหลักที่ใช้ | ฟังก์ชันใน POC |
|------|-----------------|-----------------|
| **เกษตรกร/เจ้าของฟาร์ม** | Farmer Portal | ลงทะเบียนฟาร์ม・สัตว์, บันทึกข้อมูลประจำวัน, ขอนัดตรวจ (ปกติ/ฉุกเฉิน) |
| **สัตวแพทย์/เจ้าหน้าที่** | Examination | รับเคส, บันทึกการตรวจรักษา, ถ่าย/อัปโหลดภาพ, ลงพื้นที่ฟาร์ม (ถ้ามี) |
| **นักศึกษา** | Online Teaching | ลงทะเบียนรับเคส, ดูประวัติ, บันทึกผลปฏิบัติ (ตามสิทธิ์) |
| **อาจารย์สัตวแพทย์** | Online Teaching + Examination | อนุมัติ/แก้ไข/ตรวจสอบเคส, มอบเคสให้นักศึกษา, ดูต้นทุนการสอน |

ใน POC ควรมีอย่างน้อย 4 roles นี้ และกำหนดสิทธิ์ให้ชัด (อ่าน/เขียน/อนุมัติ) เพื่อให้เห็น “ลำดับขั้น” ตามข้อ 3 ของเอกสาร

---

## 3. การวางระบบแต่ละส่วนให้ครบ Loop

### 3.1 ระบบพอร์ทัลเกษตรกร (Farmer Portal) – จุดเริ่มต้น Loop

**วัตถุประสงค์ใน POC:** ให้เกษตรกรเป็นคน “สร้างข้อมูลและขอนัด” เพื่อให้มีเคสไปต่อที่ระบบตรวจรักษาและห้องเรียน

| ฟังก์ชันจากเอกสาร | สิ่งที่ POC ควรมี (ตัวอย่าง) |
|-------------------|------------------------------|
| จัดการข้อมูลสัตว์เบื้องต้น | ลงทะเบียนฟาร์ม (มีแล้ว), ลงทะเบียนสัตว์ในฟาร์ม ( species, หมายเลข, ชื่อ, ฯลฯ ) |
| ข้อมูลประจำวัน | ฟอร์มบันทึกประจำวัน: วันที่, ปริมาณอาหาร, อาการ (ข้อความ), น้ำหนัก, ปริมาณน้ำนม (ถ้าเป็นโคนม) |
| แจ้งเตือนและนัดหมาย | ปุ่ม “ขอนัดตรวจ” (ปกติ/ฉุกเฉิน), หน้า “นัดของฉัน” แสดงสถานะ (รอ/อนุมัติ/เสร็จแล้ว) |

**ข้อมูลที่ต้องมีใน DB (แนวทาง):**

- ฟาร์ม (มีแล้ว), สัตว์ (Animal: ฟาร์ม, ชนิดสัตว์, หมายเลข, ชื่อ, ฯลฯ )
- บันทึกประจำวัน (DailyRecord: สัตว์, วันที่, อาหาร, อาการ, น้ำหนัก, นม ฯลฯ )
- นัดหมาย (Appointment: ฟาร์ม/สัตว์, ประเภทนัด, สถานะ, วันที่นัด, ฯลฯ )

**ผลลัพธ์ของ Loop:** เกษตรกรสร้าง “สัตว์” และ “นัด” → ระบบมีเคสให้แผนกตรวจรักษาและห้องเรียนใช้ต่อ

---

### 3.2 ระบบการตรวจรักษา (Examination & Farm) – จุดกลาง Loop

**วัตถุประสงค์ใน POC:** รับเคสจากนัด/ฟาร์ม → บันทึกการตรวจรักษา → ถ่ายภาพอาการ → (ถ้าต้องการ) ผูกกับเคสการสอน

| ฟังก์ชันจากเอกสาร | สิ่งที่ POC ควรมี (ตัวอย่าง) |
|-------------------|------------------------------|
| หน้าจอแสดงผล/บันทึกการตรวจรักษา | หน้ารายการเคส (จากนัด), แยกตามคลินิก/ประเภท (เช่น ปศุสัตว์, ม้า) ได้ |
| ถ่ายภาพและอัปโหลด | บันทึกการตรวจ 1 รายการ มีได้หลายรูป (อัปโหลดไฟล์ภาพ), แสดงในหน้ารายละเอียดการตรวจ |
| ลงพื้นที่ฟาร์ม | ประเภทการตรวจ: “ที่คลินิก” หรือ “ลงพื้นที่ฟาร์ม” (ผูกกับฟาร์ม + สัตว์) |

**ข้อมูลที่ต้องมีใน DB (แนวทาง):**

- การตรวจรักษา (Examination ขยายหรือใหม่): ผูกกับ Appointment/Animal/Farm, คลินิก/ประเภท, วันที่, สรุปผล, สถานะ (รออาจารย์อนุมัติ/อนุมัติแล้ว ฯลฯ )
- ภาพประกอบ (ExaminationImage: การตรวจ 1 รายการ – หลายรูป, path/url)
- (ถ้าต้องการ) การลงพื้นที่ฟาร์ม: ใช้ฟิลด์ประเภทการตรวจ + ฟาร์ม ที่มีอยู่แล้ว

**ผลลัพธ์ของ Loop:** แต่ละเคสมีการตรวจรักษาที่มีประวัติ + ภาพ → สามารถ “มอบให้นักศึกษา” ในระบบห้องเรียนและบันทึกต้นทุนได้

---

### 3.3 ระบบห้องเรียนออนไลน์ (Online Teaching Classroom) – จุดควบคุมและสอน

**วัตถุประสงค์ใน POC:** แสดง “ลำดับขั้นสิทธิ์” และ “ต้นทุนการศึกษา” ตามเอกสาร

| ฟังก์ชันจากเอกสาร | สิ่งที่ POC ควรมี (ตัวอย่าง) |
|-------------------|------------------------------|
| การจัดการสิทธิ์แบบลำดับขั้น | นักศึกษา: ลงทะเบียนรับเคส, ดูประวัติ, บันทึกผล (แบบที่อาจารย์ต้องอนุมัติ) |
| | อาจารย์: อนุมัติเคสที่นักศึกษาบันทึก, แก้ไข, ตรวจสอบ และปิดเคส |
| การคำนวณต้นทุนการศึกษา | บันทึก “ต้นทุนการรักษาจริง” ต่อเคส (แม้ยกเว้นค่าใช้จ่ายจากเจ้าของ) เพื่อใช้รายงานต้นทุนการสอน |

**ข้อมูลที่ต้องมีใน DB (แนวทาง):**

- การมอบเคสให้นักศึกษา (TeachingCase: การตรวจ 1 รายการ – นักศึกษา 1 คน, สถานะ: รออนุมัติ/อนุมัติแล้ว)
- ต้นทุนการสอน (EducationCost หรือฟิลด์ใน TeachingCase/Examination: ต้นทุนจริง, สถานะการเก็บเงินจากเจ้าของ เช่น ยกเว้นเพื่อการสอน)

**ผลลัพธ์ของ Loop:** เคสจากเกษตรกร → ผ่านการตรวจรักษา (และภาพ) → นักศึกษาฝึกปฏิบัติ → อาจารย์อนุมัติ → ระบบบันทึกต้นทุนจริง → POC แสดงครบทั้งการให้บริการและมิติการสอน

## 4. ลำดับการพัฒนาที่แนะนำ (Phased เพื่อให้เห็น Loop เร็ว)

เพื่อให้ “เห็น loop การใช้งานเบื้องต้น” โดยไม่ต้องทำทุกฟีเจอร์พร้อมกัน แนะนำเป็น 2 ช่วง:

### Phase 1: Loop พื้นฐาน (เห็นครบวงจร)

1. **Farmer Portal (ขั้นต่ำ)**  
   - ลงทะเบียนสัตว์ในฟาร์ม  
   - บันทึกข้อมูลประจำวัน (อย่างน้อย: วันที่, อาการ, น้ำหนัก)  
   - ขอนัดตรวจ (ประเภท: ปกติ/ฉุกเฉิน), ดูรายการนัดของฉัน  

2. **Examination (ขั้นต่ำ)**  
   - หน้ารายการนัด (ดึงจาก Appointment)  
   - เปิดเคส → บันทึกการตรวจรักษา (สรุป, ประเภทคลินิก, ผูกกับสัตว์/ฟาร์ม)  
   - อัปโหลดภาพประกอบการตรวจ 1 รายการ (หลายรูปได้)  

3. **Online Teaching (ขั้นต่ำ)**  
   - อาจารย์ “มอบเคส” ให้นักศึกษา (เลือกจากการตรวจที่ยังไม่ปิด)  
   - นักศึกษาเห็นเคสที่ได้รับมอบ, ดูประวัติ, บันทึกผล (สถานะรออนุมัติ)  
   - อาจารย์อนุมัติ/ส่งกลับแก้ไข  
   - บันทึกต้นทุนการรักษาจริงต่อเคส (จำนวนเงิน + หมายเหตุ เช่น “ยกเว้นเพื่อการสอน”)  

4. **ปิด Loop**  
   - หลังอาจารย์อนุมัติ/ปิดเคส → อัปเดตสถานะนัดเป็น “เสร็จแล้ว” (และถ้ามี: แสดงคำแนะนำหรือนัดครั้งถัดไปในพอร์ทัลเกษตรกร)  

ใน Phase 1 ยังไม่จำเป็นต้องมี “แจ้งเตือนโรคระบาด” หรือ “การส่งต่อ Referral” ก็ได้ เพื่อโฟกัสให้ loop หลักชัดเจน

### Phase 2: เพิ่มความสมบูรณ์ (หลัง POC Phase 1 ใช้งานได้)

- Farmer: แจ้งเตือนในพื้นที่ (ข้อความ/ประกาศ), หน้าประวัติสัตว์แบบย่อ  
- Examination: แยกหน้าตามคลินิก, ลงพื้นที่ฟาร์ม (เลือกฟาร์มจากนัด)  
- Teaching: รายงานสรุปต้นทุนการสอน, สิทธิ์การแก้ไขแบบละเอียด

## 5. โครงสร้างข้อมูลที่แนะนำ (ระดับ Entity สำหรับ POC)

```
Farm (มีแล้ว)
Animal (ฟาร์ม, ชนิด, หมายเลข, ชื่อ, ...)
DailyRecord (สัตว์, วันที่, อาหาร, อาการ, น้ำหนัก, นม, ...)
Appointment (ฟาร์ม/สัตว์, ประเภทนัด, สถานะ, วันที่นัด, ...)

Examination (ขยาย/ปรับ: นัด, สัตว์, ฟาร์ม, ประเภทคลินิก, วันที่, สรุป, สถานะ)
ExaminationImage (การตรวจ, path/url, ลำดับ)

TeachingCase (การตรวจ, นักศึกษา, สถานะ: รออนุมัติ/อนุมัติ, ต้นทุนจริง, หมายเหตุ)
User (ถ้ายังไม่มี: บทบาท = เกษตรกร/สัตวแพทย์/นักศึกษา/อาจารย์)
```

การเชื่อมกับระบบเดิม (เช่น HN, ฐานข้อมูลคนละที่) สามารถทำใน Phase 2 หรือหลัง POC ได้

## 6. สรุป: การวางระบบเพื่อให้ POC ครบ Loop

| เป้าหมาย | วิธีวางระบบ |
|----------|----------------|
| **ครบ loop** | เกษตรกรสร้างนัด → โรงพยาบาลรับเคสและบันทึกการตรวจ+ภาพ → ห้องเรียนมอบเคสให้นักศึกษา อาจารย์อนุมัติ และบันทึกต้นทุน → อัปเดตสถานะนัดและแสดงผลกลับเกษตรกร |
| **เห็นรูปแบบการใช้งานเบื้องต้น** | ใช้ 4 roles (เกษตรกร, สัตวแพทย์, นักศึกษา, อาจารย์) และฟังก์ชันขั้นต่ำในตารางด้านบน ใน Phase 1 |
| **ไม่ทำครั้งเดียวทั้งหมด** | ทำ Phase 1 ให้ครบ loop ก่อน แล้วค่อย Phase 2 เพิ่มแจ้งเตือน, รายงาน, สิทธิ์ละเอียด |
| **สอดคล้องเอกสาร req** | ระบบย่อย 2, 3, 4 ทำงานร่วมกันได้ และแสดงทั้งการตรวจรักษา (รวมภาพและลงพื้นที่), การสอนแบบลำดับขั้น และพอร์ทัลเกษตรกร (ข้อมูลสัตว์ + นัด) |

ถ้าต้องการให้ช่วยลงรายละเอียดเป็น **รายการ Task/User Story** หรือ **Schema Prisma/ตาราง** สำหรับ Phase 1 สามารถระบุได้ว่าจะเริ่มจากส่วนไหน (Farmer / Examination / Teaching) ก่อน
